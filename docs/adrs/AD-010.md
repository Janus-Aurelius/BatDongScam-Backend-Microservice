# AD-010: Adopt Circuit Breaker and Retry Patterns for External Integrations

*   **Status:** Proposed
*   **Context:** The macroservice relies on the legacy monolith and external APIs (Cloudinary, PayPal/PayOS). If the legacy monolith slows down, threads in the new macroservice will hang, leading to a cascading failure that could crash the entire system.
*   **Decision:** Use **Resilience4j** to wrap all external calls (HTTP/Feign) with Circuit Breakers, Rate Limiters, and Retries.
*   **Consequences (Positive):**
    *   **Fault Tolerance:** If a dependency is down, the Circuit Breaker "opens," returning a fallback response immediately instead of wasting resources.
    *   **Self-Healing:** Retries with exponential backoff allow transient network issues to resolve without manual intervention.
*   **Consequences (Negative/Risks & Mitigation):**
    *   **Complexity:** Managing fallback logic for every external call. *Mitigation:* Define global fallback strategies (e.g., returning cached data or a generic "Service Temporarily Unavailable" message).
