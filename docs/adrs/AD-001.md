# AD-001: Transition to a Distributed Microservices Architecture

*   **Status:** Accepted
*   **Context:** Our current `legacy/` codebase is a modular monolith. While it follows Spring Boot best practices, it suffers from "God Classes" (`Property`, `User`) that tightly couple the Property Catalog, Identity Management, and Financial domains. This coupling hinders independent scaling—for instance, the high-traffic Search and Analytics logic (currently in `SearchServiceImpl`) competes for resources with critical transactional processes like Contract execution.
*   **Decision:** We will adopt a Microservices architecture using the **Strangler Fig Pattern**. We are initiating this by building a hexagonal "Macroservice" (`bds-core-macroservice`) that will eventually be decomposed into granular services (Identity, Property, Transaction, Financial, and Search).
*   **Consequences (Positive):** 
    *   **Domain Isolation:** Clearer boundaries for complex domains like `contract` and `payment`.
    *   **Independent Scaling:** High-read services (Search/Analytics) can be scaled horizontally without duplicating the entire transactional engine.
    *   **Technology Flexibility:** Enables optimized tech stacks for different workloads (e.g., specific libraries for OCR or AI valuation).
*   **Consequences (Negative/Risks & Mitigation):** 
    *   **Operational Complexity:** Managing multiple deployment units. *Mitigation:* We are introducing a centralized infrastructure phase (Service Registry, Config Server) as outlined in our migration roadmap.
    *   **Distributed Transactions:** Replacing JPA `@ManyToOne` fetches with inter-service calls. *Mitigation:* Moving toward eventual consistency and using `propertyId`/`userId` UUIDs instead of direct entity references.
