# AD-003: Adopt Stateless JWT-based Authentication

*   **Status:** Accepted
*   **Context:** The legacy system already utilizes `jjwt` and `SessionCreationPolicy.STATELESS` in `WebSecurityConfig.java`. This is a significant advantage for microservices, as it avoids the need for "sticky sessions" or a centralized session store across the distributed cluster.
*   **Decision:** Continue and formalize the use of stateless JSON Web Tokens (JWT) for authentication and authorization.
*   **Consequences (Positive):**
    *   **Scalability:** Services can verify tokens locally using a public key/shared secret without querying an Auth service for every request.
    *   **Mobile-Friendly:** Well-suited for the platform's mobile client requirements.
*   **Consequences (Negative/Risks & Mitigation):**
    *   **Token Revocation:** Stateless tokens cannot be easily revoked before expiration. *Mitigation:* Use short-lived Access Tokens and implement a Refresh Token lifecycle (leveraging `RefreshTokenExpiredException` logic already present in the codebase).
    *   **Payload Size:** Large JWTs can increase overhead. *Mitigation:* Keep claims minimal; only include essential UUIDs and roles.
