# AD-002: Implement API Gateway as the Single Entry Point

*   **Status:** Accepted
*   **Context:** As we decompose the `legacy/` monolith, clients (Web/Mobile) would otherwise need to track multiple service endpoints. Furthermore, cross-cutting concerns like CORS, rate limiting, and global exception handling are currently duplicated or manually managed in `WebMvcConfig.java` and `AppExceptionHandler.java`.
*   **Decision:** Implement an API Gateway (using Spring Cloud Gateway) to act as the unified interface for all external traffic.
*   **Consequences (Positive):**
    *   **Centralized Security:** Consolidates SSL termination and CORS configuration in one place.
    *   **Protocol Translation:** Allows internal services to use gRPC while exposing REST to clients.
    *   **Routing Logic:** Enables seamless migration by routing specific paths (e.g., `/api/v1/search/**`) to new microservices while defaulting others to the legacy monolith.
*   **Consequences (Negative/Risks & Mitigation):**
    *   **Single Point of Failure:** If the gateway goes down, the entire system is inaccessible. *Mitigation:* Deploy the gateway in a highly available cluster with load balancing.
    *   **Performance Bottleneck:** Extra hop for every request. *Mitigation:* Ensure the gateway is reactive (Netty-based) and avoid heavy logic/processing within the gateway layer.
