# AD-012: Enforce Logical Database Separation for Extracted Services

*   **Status:** Proposed
*   **Context:** Currently, the `bds-core-macroservice` and the `legacy` monolith share the same PostgreSQL/MongoDB instances (even if using different tables/collections). This prevents true independent scaling and risks accidental cross-service schema coupling.
*   **Decision:** As we transition from the Macroservice to discrete Microservices, each service MUST move to its own logical database (separate schema in Postgres, separate DB in Mongo). Data from other domains must be accessed via API calls or event replication, NEVER via direct SQL joins across schemas.
*   **Consequences (Positive):**
    *   **True Autonomy:** Teams can change their service's database schema without coordinating with other teams.
    *   **Scalability:** Each database can be tuned or scaled independently (e.g., different RDS instances).
*   **Consequences (Negative/Risks & Mitigation):**
    *   **Join Complexity:** Cross-domain reporting becomes harder. *Mitigation:* Use the "API Composition" pattern in the Gateway or leverage the Search/Analytics service (AD-006) for aggregated views.
