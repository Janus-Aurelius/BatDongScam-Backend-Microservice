# AD-004: Implement Redis Pub/Sub for Asynchronous Event-Driven Communication

*   **Status:** Accepted
*   **Context:** Currently, the system uses Spring's `@Async` for tasks like search logging and dummy data generation. In a distributed environment, memory-based async calls are unreliable; if a node restarts, the task is lost. We need a way to decouple services, such as notifying the Search service when a `PropertyCreatedEvent` occurs in the Core domain.
*   **Decision:** Use Redis Pub/Sub as the primary mechanism for internal asynchronous event broadcasting between services.
*   **Consequences (Positive):**
    *   **Loose Coupling:** The Property service doesn't need to know about the Search or Analytics services.
    *   **Performance:** Offloads heavy tasks (like updating MongoDB search indices) from the main request thread.
*   **Consequences (Negative/Risks & Mitigation):**
    *   **At-most-once Delivery:** Redis Pub/Sub does not guarantee message persistence if a subscriber is offline. *Mitigation:* For critical events (like `PaymentSucceeded`), we will implement an "Outbox Pattern" or consider migrating to Redis Streams/Kafka for guaranteed delivery as specified in the migration plan.
    *   **Traceability:** Harder to debug distributed flows. *Mitigation:* Implement Correlation IDs in the `bds-common` event DTOs.
