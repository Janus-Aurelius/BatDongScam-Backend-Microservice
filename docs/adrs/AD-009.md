# AD-009: Use Transactional Outbox Pattern for Reliable Event Delivery

*   **Status:** Proposed
*   **Context:** AD-004 specifies Redis Pub/Sub for events, which is "at-most-once." If the database transaction for a `Contract` succeeds but the Redis publish fails (or the network hiccups), the Search or Payment services will never know. This leads to data inconsistency.
*   **Decision:** We will implement the Transactional Outbox pattern. Instead of publishing to Redis directly in the service logic, we will write the event to a `messages` table in the local PostgreSQL database as part of the same transaction. A background "Relay" process will then poll this table and publish the events to Redis/Kafka.
*   **Consequences (Positive):**
    *   **Guaranteed Delivery:** Events are only sent if the database transaction succeeds.
    *   **Consistency:** Eliminates the "Ghost Update" problem where DB and Event Bus fall out of sync.
*   **Consequences (Negative/Risks & Mitigation):**
    *   **Database Load:** Additional writes/reads for the outbox table. *Mitigation:* Use an efficient indexing strategy and prune processed messages regularly.
