# AD-006: Standardize on PostgreSQL for Transactions and MongoDB for Analytics/Search

*   **Status:** Accepted
*   **Context:** Our 40+ JPA `@Entity` tables (for core workflows) and 28+ MongoDB `@Document` collections (for search/reporting) represent a mature data foundation. We need to maintain this split while ensuring data integrity across the service boundaries.
*   **Decision:** Standardize PostgreSQL as the source of truth for all transactional data and MongoDB for the read-optimized Search and Analytics domains.
*   **Consequences (Positive):**
    *   **Transactional Integrity:** PostgreSQL ensures that Property-Contract-Payment relations remain consistent via ACID properties.
    *   **Query Flexibility:** MongoDB allows for rapid aggregation of "Owner Insights" and "Sales Reports" without impacting the performance of the transactional DB.
*   **Consequences (Negative/Risks & Mitigation):**
    *   **Data Synchronization:** Risk of the Search index (MongoDB) falling out of sync with the Property data (PostgreSQL). *Mitigation:* Use the event-driven architecture (AD-004) to trigger MongoDB updates immediately upon PostgreSQL commits.
    *   **Governance:** Need for clear ownership of shared data like `Ward`/`District`/`City`. *Mitigation:* The Property service will act as the master for Location data, providing it to others via APIs or replicated events.
