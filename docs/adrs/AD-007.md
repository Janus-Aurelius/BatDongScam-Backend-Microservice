# AD-007: Use Facades to Enforce Internal Module Boundaries

*   **Status:** Accepted
*   **Context:** In the `bds-core-macroservice`, we have multiple domain modules (e.g., `property`, `transaction`). To prevent tight coupling and circular dependencies that plagued the legacy monolith, we need a strict protocol for cross-module communication.
*   **Decision:** Each domain module must expose a single `Facade` interface (e.g., `PropertyFacade.java`) located in its `api` package. This Facade is the **only** entry point for other modules. Direct access to a module's internal services, repositories, or domain entities from outside the module is strictly forbidden.
*   **Consequences (Positive):**
    *   **Loose Coupling:** Changes to the internal logic of the `property` module do not break the `transaction` module as long as the Facade contract remains stable.
    *   **Easier Extraction:** Since all interactions are routed through a Facade, extracting a module into its own microservice in the future becomes a matter of replacing the local Facade implementation with a REST or gRPC client.
*   **Consequences (Negative/Risks & Mitigation):**
    *   **Boilerplate:** Requires mapping internal entities to shared DTOs/Snapshots. *Mitigation:* Use MapStruct for efficient, type-safe mapping.
    *   **Facade Bloat:** The Facade can become a "God Interface." *Mitigation:* Periodically review the Facade and split into specialized ports if it grows too large.
