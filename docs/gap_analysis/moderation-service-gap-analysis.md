# Post-Merge Gap Analysis Report: `moderation-service`

## Executive Summary
During this post-merge synchronization phase, the [moderation-service](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service) was analyzed. The service currently acts as an isolated, standalone Spring Boot application rather than a fully integrated, secure microservice in the BatDongSan microservices architecture.

Although the database layer of [moderation-service](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service) has been refactored for a decoupled environment—successfully replacing strict JPA entity relationships with raw UUID fields (such as `reporterId` and `relatedEntityId`) and replacing joint media tables with `@ElementCollection` of `evidenceUrls` in [ViolationReport](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/moderation/entity/ViolationReport.java)—the service suffers from several critical architectural, integration, and security gaps:
1. **Critical Security Bypasses:** The authorization configuration in [SecurityConfig](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/security/SecurityConfig.java#L22) permits all requests (`.anyRequest().permitAll()`), and [HeaderUserIdAuthenticationFilter](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/security/HeaderUserIdAuthenticationFilter.java#L28) leaves role authorities empty. This exposes admin-only endpoints to unauthorized access.
2. **Missing Inter-Service Integration:** The service has no integration layer (neither REST clients like Feign/WebClient nor Kafka consumer/producer setup). As a result, critical cross-module validations (such as validating if a reported property or user actually exists) and data-enrichment tasks (fetching names and avatars for reports) are entirely missing or skipped.
3. **Toothless Penalty Enforcement:** Admin actions, such as applying penalties (e.g. suspending accounts, removing posts), are saved locally in the PostgreSQL database but are never propagated to the core system because of the complete absence of event-driven communication (Kafka).
4. **Contract and Build Isolation:** The service duplicates DTO wrappers and models rather than importing standard constructs from the shared [bds-common](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/bds-common) module. Furthermore, it is not registered as a module in the root [pom.xml](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/pom.xml).

In the new service, this validation is completely bypassed (noted as `// TODO: Phase 2 - Call Core Service via HTTP to validate ID` on [ViolationServiceImpl.java:L138](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/moderation/service/impl/ViolationServiceImpl.java#L138)).

  The daily and monthly reporting scheduler [ViolationReportScheduler.java](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/legacy/src/main/java/com/se100/bds/services/domains/report/scheduler/ViolationReportScheduler.java) which saves monthly aggregations (cumulative violation counts, average resolution times, account suspensions, and removed posts count) to a MongoDB reporting database was completely left behind and not migrated to the new service.
* **Evidence Media Classification:**
  In legacy, evidence files are classified into specific media types (`IMAGE` or `DOCUMENT`) using `Constants.MediaTypeEnum` in [ViolationServiceImpl.java:L229-L243](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/legacy/src/main/java/com/se100/bds/services/domains/violation/impl/ViolationServiceImpl.java#L229-L243). * **Mocked Storage Integration:**
  While legacy uploads files to a live CDN via `CloudinaryService` ([ViolationServiceImpl.java:L228](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/legacy/src/main/java/com/se100/bds/services/domains/violation/impl/ViolationServiceImpl.java#L228)), the target service uses [MockFileStorageService.java](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/storage/MockFileStorageService.java) which stub uploads and prefixes URLs with `mock://`, which will fail in a staging or production environment.

## Integration Gaps (Core Macroservice)
The target service does not integrate with the primary [bds-core-macroservice](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/bds-core-macroservice), causing data consistency and visual rendering problems:

* **DTO Data Enrichment Deficit (Missing Details):**
  In the legacy monolith, the response models [ViolationAdminItem](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/legacy/src/main/java/com/se100/bds/dtos/responses/violation/ViolationAdminItem.java), [ViolationAdminDetails](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/legacy/src/main/java/com/se100/bds/dtos/responses/violation/ViolationAdminDetails.java), [ViolationUserDetails](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/legacy/src/main/java/com/se100/bds/dtos/responses/violation/ViolationUserDetails.java), and [ViolationUserItem](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/legacy/src/main/java/com/se100/bds/dtos/responses/violation/ViolationUserItem.java) are enriched with details from other contexts (e.g. `reporterName`, `reportedName`, `reporterAvatarUrl`, `reportedAvatarUrl`, `targetName`, `userTier`, property details).
  In the new [moderation-service](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service), because there is no API client, the DTOs only expose raw UUIDs, leaving fields like `reporterName` or `reportedName` completely missing. This leaves the admin panel and user profile pages without names or avatar URLs.
* **Skipped Reporter Name Filtering:**
  In the admin violation list endpoint, sorting or filtering by reporter `name` is simply bypassed and logged in [ViolationServiceImpl.java:L73-L76](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/moderation/service/impl/ViolationServiceImpl.java#L73-L76). In the legacy monolith, this was done using a SQL join on the User table.
* **Lack of Kafka Event Synchronization for Penalties:**
  When an admin updates a report status to resolved and applies a penalty (like `REMOVED_POST` or `SUSPENDED_ACCOUNT`), the core service must execute this decision (suspending the user, removing the listing). However, [ViolationServiceImpl.java:L177-L205](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/moderation/service/impl/ViolationServiceImpl.java#L177-L205) only saves this status locally. It does not publish events (e.g., `violation-penalty-applied`) to a Kafka topic, meaning the penalties never take effect on actual properties or user accounts.
* **Missing Project Module Registration:**
  The parent [pom.xml](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/pom.xml#L18) lists `bds-core-macroservice` and `bds-common` as modules, but completely ignores `moderation-service`. It must be registered as a Maven module to enable consolidated builds.

## Contract Violations

* **POM Dependency Absence:**
  The target service's [pom.xml](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/pom.xml) does not reference the shared [bds-common](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/bds-common) dependency and is not configured with the platform's parent POM `batdongsan-platform`.
* **Duplicated DTO Base Classes:**
  The class [AbstractBaseDataResponse](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/common/model/AbstractBaseDataResponse.java) is duplicated locally within the service package instead of being imported from the shared [com.se.bds.common.dto.AbstractBaseDataResponse](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/bds-common/src/main/java/com/se/bds/common/dto/AbstractBaseDataResponse.java).
* **Non-Standard API Response Wrappers:**
  The controllers in [moderation-service](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service) wrap payloads in custom response objects:
  - [AbstractBaseResponse](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/api/response/AbstractBaseResponse.java)
  - [SingleResponse](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/api/response/SingleResponse.java)
  - [PageResponse](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/api/response/PageResponse.java)
  These custom responses violate standard contracts. They should be refactored to use [com.se.bds.common.dto.ApiResponse](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/bds-common/src/main/java/com/se/bds/common/dto/ApiResponse.java).

* **Local Enums Duplication:**
  Enums such as `ViolationReportedTypeEnum`, `ViolationTypeEnum`, `ViolationStatusEnum`, and `PenaltyAppliedEnum` are defined locally in [Constants.java](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/common/Constants.java). To enable proper type-safety in inter-service Kafka communications, these enums should be refactored and extracted to [bds-common](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/bds-common) so both the moderation and core services can share them.

## Communication Anti-patterns

* **Unprotected Administrative Endpoints (PermitAll):**
  In [SecurityConfig.java](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/security/SecurityConfig.java#L22), the filter chain is configured with `.anyRequest().permitAll()`. Admin paths like `/violations/admin/**` have no role checking. This allows any user (or even unauthenticated requests) to read, modify, and delete violation reports.
* **Empty Authority Roles in Security context:**
  [HeaderUserIdAuthenticationFilter.java](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/security/HeaderUserIdAuthenticationFilter.java#L28) extracts the `X-User-Id` header but instantiates the token with `List.of()` authorities. This prevents the usage of role-based annotations like `@PreAuthorize("hasRole('ADMIN')")` in the controllers.
* **Missing Gateway JWT Role Propagation:**
  The gateway filter [JwtAuthenticationFilter.java](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/api-gateway/src/main/java/com/se100/bds/gateway/filter/JwtAuthenticationFilter.java#L57) only extracts the `userId` claim and forwards it as the `X-User-Id` header to downstream services. It does not extract or forward the user's role claim (e.g. `X-User-Roles`), which is necessary for downstream services to enforce role-based access control (RBAC).
* **Missing Cloud Service Registry (Eureka Client):**
  There is no Eureka client configuration in `moderation-service`. Without registry integration, the [api-gateway](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/api-gateway) cannot discover this service dynamically, making it unreachable through the gateway routes.
* **Lack of Resilient Communication Client:**
  No communication channels (OpenFeign or WebClient) are configured. If implemented synchronously, these client calls must be guarded by proper timeouts, retries, and circuit breakers (Resilience4j) to prevent cascading failures in case the core service goes down.

- [ ] **Standardize DTOs:** Replace local definitions of [AbstractBaseDataResponse](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/moderation-service/src/main/java/microservices/moderationservice/common/model/AbstractBaseDataResponse.java) with imports from [com.se.bds.common.dto.AbstractBaseDataResponse](file:///home/annguyen/master_projects/sem2_year3_projects/BatDongSan/BatDongScam-Backend-Microservice/bds-common/src/main/java/com/se/bds/common/dto/AbstractBaseDataResponse.java).

- [ ] **Add Eureka Client:** Add the `spring-cloud-starter-netflix-eureka-client` dependency to `moderation-service/pom.xml` and enable discovery in `application.properties`.
